# Contributing — GlowMath Course

> **Last updated:** 2026-05-28. How to contribute. Style + naming + workflow.
>
> Project is currently **solo-dev** on `main`. Multi-dev / PR conventions below activate when a second contributor joins.

## Code style

### Python

- Follow **PEP 8** (run `python -m py_compile` mentally — if it would pass `flake8 --max-line-length=100`, it's fine)
- Line length: **100 characters** (relaxed from PEP 8's 79)
- Indentation: 4 spaces, never tabs
- Use **f-strings** for formatting (not `.format()` or `%`)
- Import order: stdlib → 3rd-party → Django → local apps. Blank line between groups.

```python
# Good
import os
from datetime import datetime, timedelta
from decimal import Decimal

from django.contrib.auth.decorators import login_required
from django.shortcuts import get_object_or_404, render

from accounts.decorators import role_required
from accounts.models import User
from enrollments.models import Enrollment, EnrollmentStatus
```

### Naming conventions

#### Python

| Element | Convention | Example |
|---|---|---|
| Classes | PascalCase | `StudentProfile`, `TeacherRating` |
| Functions / methods | snake_case | `get_active_classes`, `mark_attendance` |
| Variables | snake_case | `enrollment_count`, `student_profile` |
| Constants | UPPER_SNAKE_CASE | `ENABLE_PAYMENT_FEATURE`, `LOGIN_URL` |
| Private | `_leading_underscore` | `_try_enroll`, `_invalidate_cache` |

#### Django models

- Names: singular (`Kelas`, `Enrollment` — NOT `Classes`, `Enrollments`)
- Use **`Kelas`** NOT `Class` (`class` is a Python reserved word)
- Related names: plural snake_case (`related_name='enrollments'`, `related_name='ratings'`)
- Always define `__str__` for admin display
- Always define `class Meta: ordering = [...]` so admin and querysets have a default

#### URLs

- URL pattern names: snake_case (`browse_classes`, `my_schedule`, `rate_teacher`)
- App namespaces: lowercase, match the app folder (`accounts:`, `academics:`)
- Always namespace `{% url %}` calls: `{% url 'academics:class_browse' %}` — never bare `{% url 'class_browse' %}`
- **Verify URL names against [URL_ROUTES.md](URL_ROUTES.md) before using** — the #1 cause of `NoReverseMatch` is a guessed name.

#### Templates / UI

- **Visible strings** in Bahasa Indonesia: "Kelas Saya", "Pengaturan", "Beri Rating"
- **HTML IDs / classes** in English: `id="class-card"`, `class="sidebar-item"`
- **HTML comments** in English (developer-facing): `<!-- Sidebar Khan Playful — student only -->`

## Django conventions

### Views

- Use **function-based views** by default — they're simpler and the team is comfortable with them
- Use class-based only when inheriting a generic view simplifies the code meaningfully
- Always decorate with `@login_required` when auth is required
- Role-gate with `@role_required('STUDENT')` (or `'TEACHER'` / `'ADMIN'`) from `accounts.decorators`
- For POST-only views: `@require_POST` from `django.views.decorators.http`

### Models

- Always define `__str__`
- Always define `Meta.ordering`
- Use Indonesian `verbose_name` / `verbose_name_plural` when displayed in admin
- Use `help_text` on non-obvious fields
- Use `TextChoices` for every enum (never raw `choices=[(...)]`)
- For soft-deletable models: add `is_deleted` + `deleted_at` + a `soft_delete()` method

### Templates

- Extend `base.html` for authenticated pages
- Extend `base_auth.html` for login / register / password reset
- Extend `base_print.html` for print-friendly pages
- **Multi-line comments:** use `{% comment %} ... {% endcomment %}`. NEVER use multi-line `{# ... #}` — Django comments are single-line only and the text leaks. See [PITFALLS.md](PITFALLS.md#-multi-line---comments-leak-as-visible-text).
- **Avoid inline `<script>` blocks** for non-trivial logic. Use Alpine.js attributes (`x-data`, `x-show`, `@click`) for interaction; external file for shared scripts.
- **Always include `{% csrf_token %}`** in every POST form.

### Migrations

- Always review autogenerated migrations before applying
- Name custom migrations descriptively (e.g., `0005_backfill_subject_icons.py`)
- **Don't edit migrations that have been applied** — create a new one to amend

### Caching

- Cache keys use `f'<entity>_{id}_<field>'` format — e.g., `f'sidebar_pending_ratings_{user_id}'`
- Invalidate via signals on `post_save` + `post_delete`
- Default cache backend is `LocMemCache`; safe to use across dev

## Commit message format

We use conventional-commits-style prefixes (see recent `git log`):

```
<type>(<scope optional>): <subject in present tense>

<body — optional, wrap at 72 chars>

<footer — optional>
```

| Type | Use for |
|---|---|
| `feat` | new feature |
| `fix` | bug fix |
| `docs` | documentation only |
| `style` | formatting, whitespace; no logic change |
| `refactor` | code change that's neither feat nor fix |
| `perf` | performance improvement |
| `test` | adding/updating tests |
| `chore` | maintenance (deps, config, mgmt commands) |

Examples:

```
feat(ratings): Add Rate Teacher form with interactive 5-star UI

- Alpine.js controlled star selection (hover preview + click commit)
- Suggestion chips append to textarea (8 canned phrases per axis)
- Single form submits TeacherRating + ClassRating atomically
- Cache invalidation post-submit via post_save signal
```

```
fix(sidebar): Stop multi-line {# ... #} comment from leaking as visible text

Replaced invalid Django syntax with {% comment %}...{% endcomment %}.
Affected: templates/partials/_nav_student.html.
See PITFALLS.md for the rule.
```

```
docs: Add Tier 3 documentation (DEPLOYMENT, TESTING_GUIDE, GLOSSARY, CONTRIBUTING)
```

## Branch strategy

**Currently:**
- `main` — only branch in use. Solo-dev workflow.

**When a second contributor joins, switch to:**
- `main` — production-ready (Phase 4 onward)
- `dev` — active development trunk
- `feature/<short-name>` — per-feature branches off `dev`
- Feature branch → PR into `dev` → release-time merge `dev` → `main`

Until then, commit directly to `main` is acceptable but **every commit must pass the pre-commit checklist**.

---

## Pre-commit checklist

Before every `git commit`:

- [ ] `python manage.py check` — 0 errors, 0 silenced
- [ ] Manual QA on every page touched (see [TESTING_GUIDE.md](TESTING_GUIDE.md))
- [ ] `git diff` review — no `print()` debug statements, no commented-out code blocks, no leftover `console.log`
- [ ] Update `CHANGELOG.md` if the change is user-visible or architecturally significant
- [ ] Update relevant docs:
  - New URL route → `URL_ROUTES.md`
  - New bug discovered → `PITFALLS.md`
  - New test user / scenario → `TEST_USERS.md`
  - Phase status change → `PHASE_ROADMAP.md`

---

## Dependency management

- **`requirements.txt`** is the lock file — pin every version
- Add new deps with `pip install <pkg>` then `pip freeze > requirements.txt`
- **Prefer CDN** for frontend deps (Tailwind, HTMX, Alpine.js, Tabler icons) — no npm, no build step
- Before adding a backend dep, justify why a Django built-in or existing dep can't do the job
- Keep `Faker` listed as optional — don't `import faker` outside `populate_full_demo` (which gracefully handles `ImportError`)

## Pull request flow (future multi-dev)

When PRs are introduced:

1. Branch from `dev` — `git checkout -b feature/<short-name>`
2. Make changes; commit often with conventional-commit messages
3. Run pre-commit checklist
4. Push branch: `git push -u origin feature/<short-name>`
5. Open PR into `dev` via GitHub UI
6. PR description includes: **What changed**, **Why**, **Test plan** (which routes/users you QA'd), screenshots if UI
7. Self-review (read your own diff before requesting review)
8. Address review feedback
9. Squash-merge into `dev` (preserves clean main history)

The `gh` CLI is preferred for PR creation if installed:

```bash
gh pr create --title "feat(scope): subject" --body "$(cat <<'EOF'
## What
- Bullet
- Bullet

## Why
Single sentence rationale.

## Test plan
- [ ] Manual QA on /classes/ as Rafael
- [ ] python manage.py smoke_test
- [ ] iPhone SE viewport verified
EOF
)"
```
